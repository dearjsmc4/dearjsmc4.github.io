float 이슈 해결해보기

April 1, 2019 HTML/CSS


이 기괴한 레이아웃을


이렇게 바꾸기.

요소를 정렬할 때는 flex 를 많이 쓴다고 하는데.
만약 익스플로러 하위버전까지 호환되어야 한다면 플렉스는 저멀리 안녕 하고 보내주어야 한다.
그리고 나에게는 float 나 inline 과 같은 선택지가 주어지는데,
인라인같은 경우는 요소들 사이에 공백문자가 생겨버려서 이 상황엔 적절치 않고
float 를 쓰면 정렬할 수 있다.
그런데 float 는 말 그대로 다른 요소들 위에 붕 떠있게 되기 때문에
생각 없이 쓰다보면 레이아웃이 다 망가지게 된다.

현재 박스1과 박스2에 float:left 를 준 상태이다.
float 로 정렬까지는 했는데 이 망가진 레이아웃을 어떻게 고칠 수 있을까.

1. clear: both;


박스 4에 clear:both 를 주게 되면 박스 4는 float 로부터 벗어나 그 아래에 잘 놓이게 되지만,
박스 1, 박스2 의 부모요소는 여전히 그 자식요소의 크기를 놓치고 있다.

2. overflow


박스1,2의 부모요소 에 overflow:visible 을 줬을 때.
visible 값을 주게 되면 같은 문제가 생긴다.
visible 값을 사용해서 float 이슈를 해결하려면
부모 요소에 너비와 크기를 명확히 지정해줘야 한다.
그런데 나는 박스1,2의 크기만큼 부모가 유동적으로 바뀌기를 원하기 때문에 높이값을 지정하고싶지 않다.
그럼 hidden 값을 주면?


놀랍게도 문제가 해결되었다!
내가 원하는 모양대로 잘 나와주었다.

​
예시를 위해 박스 2의 스타일을
position: relative;
left: 400px;
정도(다분히 고의적으로) 주게 되면부모요소에 주어진 overflow:hidden 속성에 충실하게도
내 박스 2가 제대로 보이지 않게 된다.
지금은 괜찮지만 나중에 문제가 생길 수도 있다.

3. CSS 가상요소

스타일 시트에 위와 같은 코드를 작성한다.

div3 은 박스1,2의 부모요소(노란색박스)의 아이디다.
::after 는 div3의 마지막에 가상의 자식노드를 생성한다.
( 반대로 ::before 는 가장 처음에 가상의 자식노드를 생성한다.)
이렇게 만들어진 가상 요소를 float 해결을 위해 쓰려면 반드시 content 가 있어야 하는데,
만약 content 에 값을 주게 되면,

​
박스1과 2 아래에 “나는 가상으로 만들어진 노드!” 라는 content 텍스트가 보이게 된다.
나는 이 가상요소에 어떤 텍스트도 보이지 않기를 원하기에 그냥 “” 로 비워두기로 했다.

그리고 이 가상요소는 날 때부터 인라인 요소인데
clear:both 는 인라인 요소에는 적용되지 않는다. 인라인블록에도 해봤는데 먹히지 않는다.
그래서 이 가상노드의 display 값을 block 으로 설정해주어서
clear:both 가 적용되도록 한다.

​
오버플로우로 인한 문제도 해결되었다.
가상요소를 사용하면 overflow:hidden 을 썼을 때 박스가 잘려보이던 문제도 해결이 된다.

git flow 사용해보기

March 20, 2019 기타

git flow 로 편리하게 브랜치를 만들고 프로젝트를 관리할 수 있다.
여러 줄의 명령이 필요한 작업을 git flow 로는 한 줄로 끝낼 수 있다.

git 설치

윈도우 사용자(나)는
Git for Windows
여기에서 git bash 를 다운받으면 자동으로 git flow 도 설치가 되어있기 때문에 편리하지만
그 외에 맥이나 리눅스 등은 설치가 따로 필요할 수 있다.
여기에 설치하는 방법이 나와있으니 참고하자.

사용

설치가 되었다면 사용할 차례만 남았다.

git flow init

레포 폴더에서 git flow init

다른 건 모르겠는데 윈도우는 이 prefix 를 설정할 때 대괄호 안이 텅 비어서 나온다.
이걸 지금 설정을 안 해주면 나중에 브랜치를 만들고 git branch 로 확인할 때
feature/브랜치이름
이렇게 나오지 않고 그냥 브랜치이름만 나온다.
그렇다고 지금 설정을 해주면 나중에 브랜치가 아예 안 만들어지고 오류가 난다.
(not a valid branch name… 이었던가)
( [feature] 이렇게 해서 그런가? [feature/] 이렇게 해야 하나?)
그래서 일단 비워두고 엔터를 눌러서 진행시켰다.

git branch 로 확인하면 master 와 develop 브랜치가 생겨있다.
git flow에서 반드시 필요한 브랜치들이다.
master 에는 사용자가 사용하게 될 완벽한 소스코드, 즉 라이브 될 것들만 존재하는 곳이다.
develop 에서 파생된 feature 브랜치에서 거의 모든 작업이 이루어지고, 작업이 완성되면 develop 으로 넣어준다.

git flow feature start

예시로 README-edit 이라는 이름을 가진 feature 브랜치를 만들어보자.

브랜치 이름은 기능이나 목적을 알아볼 수 있도록 짓는 것이 좋다.

README-edit 이라는 feature 브랜치가 만들어지면 자동으로 그 곳으로 이동이 된다.
여기에서 파일을 수정하고 add 하고 commit 하면 된다.
commit 은 동작하는 최소 단위마다 해주는 것이 좋다.
commit 을 쌓은 후에 다 됐다 싶으면 push 를 해준다.
feature 에서 push를 하지 않으면 깃헙의 네트워크에 브랜치가 보이지 않게 되니 push 를 해주자.

git flow feature finish

feature finish 를 하게 되면 develop 에 merge 가 되고,
README-edit 브랜치는 삭제되었다.
(finish 전에 push 를 해서 feature 가 존재했음을(?) 알려주는 것이 필요한 이유다.
finish 와 동시에 삭제되기 때문이다.)

내 위치는 다시 develop 으로 오게 된다.
develop에서 push를 해줘야 하긴 하지만 팀이 있다면 아마 내겐 권한이 없을 것이다(…)
모든 게 잘 완성되어서 release 를 해야 한다면 develop 에서 하면 된다.

git flow release start

git flow release start 버전명
버전명은 임의로 지었다. (0.0.1도 과분하다)
Fatal 이 뜨는 이유는 처음에 git flow init 을 했을 때 prefix 가 비워진 채 진행했기 때문이다.
별 문제 없으니 무시해도 된다.

git flow release finish

git flow release finish 버전명
으로 release 를 마치면 되는데…
사실 finish 후에 master 브랜치로 와있어야 한다는데
git branch 로 확인하면 왜 develop 에 머물러있는지 모르겠다.
강사님은 큰 문제는 아닌 것처럼 말씀하셨는데… 흠.
아무튼 중요한 건 다른 feature 들이 삭제되어있고 master 와 develop 만 남아있다는 것이다.
